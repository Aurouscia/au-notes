# 确定性

- 对于 node 依赖，**一定要**使用`npm ci`，不要使用`npm i`！，Dockerfile 中不能出现结果不确定的命令

# 排查问题的常用方法

## 查看 log

```sh
podman logs <容器名或ID>
```

- --tail N	只显示最后 N 行
- -f / --follow	实时跟踪日志输出（类似 tail -f）
- --since TIME	显示某个时间之后的日志
- --until TIME	显示某个时间之前的日志

## 启动一个临时容器

可用于测试网络等

```sh
docker run -it --rm alpine sh
```

- -it：启用交互模式并分配伪终端
- --rm：容器退出后自动删除

## 对于启动后立即退出的容器

- 可使用`--entrypoint sh`覆盖原本的 entrypoint
- 可使用`-c "sleep 3600"`让容器保持运行
- 使用`-d`避免卡住当前终端，启动容器，使用 `podman exec -it <容器名> sh`排查问题（通过内部的 shell 来手动运行可能出问题的东西）
    - 一次 thinkphp 的 services.php 生成失败，就是通过上述方法发现问题的：vendor目录属于root了而当前用户无法写入（手动运行 php think service:discover）

# 构建过程中的执行用户变动

在构建过程中，可能某个目录是 root 用户生成的，但又需要 www 用户写入文件，造成失败

注意清晰理解每一步的“当前用户”是谁，后面的步骤会不会与前面的冲突，需不需要 chmod 某些目录

USER xxx 命令就是用来切换当前用户的
