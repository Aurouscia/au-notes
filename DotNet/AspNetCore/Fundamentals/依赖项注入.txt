如果一个类要依赖另一个类，直接实例化那个类是不好的
如果要替换依赖的实现，就必须对每个依赖它的类修改类名
如果依赖项本身有依赖项，就必须在外面配置，这种配置会在各个地方反复写

应该：使用接口和基类把依赖关系抽象化
在服务容器IServiceProvider中注册依赖关系，注册发生在program.cs中
服务会自动实例化依赖项并注入到使用它的类的构造函数里，并在不需要之后释放它
定义依赖接口：
public interface IMyDependency{void WriteMessage(string message);}
由具体的依赖项实现该接口：
public class MyDependency:IMyDependency{
    public void WriteMessage(string message){
        Console.WriteLine($"The message is {message}");
    }
}
在program.cs注册：
builder.Services.AddScoped<IMyDependency,MyDependency>();
其中，addScoped指添加具有范围的服务
在需要用到的类的构造函数的参数里直接写
public IndexModel(IMyDependency mdpd){xxx;}

有一部分服务，例如(ILogger<T>)是由框架提供的，不需要注册


AddScoped:对于每个HTTP请求，对象都是同一个。
AddTransient:每次注入都是不同的对象。
AddSingleton:每次注入都是同一个对象。


和在构造函数要求依赖项一样，可以通过
this.HttpContext.RequestServices
取得服务。

如果使用Add<xxx>()添加服务，可以在构造服务的时候传入参数，例如：
var myKey = builder.Configuration["MyKey"];
builder.Services.AddSingleton<IService3>(sp => new Service3(myKey))

如果使用AddSingleton<xxx>()添加服务，对象是由容器创建的，容器会自动Dispose对象(IDisposable)。
如果使用AddSingleton(new MyService())，对象由开发者手动创建，容器不会释放对象。




MVC中，
Controller可以直接在构造函数要来Service对象。
Action可以在参数里加上[FromServices]
public IActionResult About([FromServices] IDateTime dateTime)
{
    return Content( $"Current server time: {dateTime.Now}");
}

View中可以使用
@inject StatisticsService StatsService
要来Service对象。
