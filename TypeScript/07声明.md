# declare关键字
ts中，declare关键字用于声明类型信息而不提供具体实现，告诉编译器"这个变量/函数/类存在于运行时，但我现在不定义它"。

## 示例
### 告诉 TS：运行时全局会有一个 myLib 变量
```ts
declare const myLib: {
  version: string;
  doSomething(): void;
};

// 编写时不会报错，尽管没定义实现（myLib可能在别处，例如cdn，添加到全局了）
console.log(myLib.version);  // ✅
```

### 为 JavaScript 代码提供类型
```ts
// utils.js (纯 JS，无类型)
export function add(a, b) { return a + b; }

// utils.d.ts (类型声明文件)
export function add(a: number, b: number): number;
```

**错误示例**：`.d.ts` 文件中不需要 `declare export` 这种写法
```ts
// ❌ 错误：declare export 不是正确的语法
declare export function add(a: number, b: number): number;

// ✅ 正确：直接 export 即可（在 .d.ts 文件中 declare 是可选的）
export function add(a: number, b: number): number;

// ✅ 也正确：显式使用 declare 后再 export
declare function add(a: number, b: number): number;
export { add };
```

### 为已有的模块提供/扩展类型
```ts
// types.d.ts
declare module 'legacy-lib' {
  // 这个库没有 @types 包，自己补充
  export function oldFunction(): void;
  export const VERSION: string;
}

// 使用时
import { oldFunction } from 'legacy-lib';  // ✅ 不再报错"找不到模块"
```

### 声明文件中的顶层导出
```ts
// index.d.ts
declare namespace MyLib {
  interface Config {
    timeout: number;
  }
  
  function configure(options: Config): void;
}

// 导出整个命名空间
export = MyLib;
export as namespace MyLib;  // 同时作为全局变量可用
```

### 声明环境变量
```ts
// env.d.ts
declare namespace NodeJS {
  interface ProcessEnv {
    readonly API_URL: string;
    readonly NODE_ENV: 'development' | 'production';
  }
}

// 使用时
process.env.API_URL;  // ✅ 类型是 string，且有自动补全
```

### 扩展全局类型
```ts
declare global {
  interface Window {
    __REDUX_DEVTOOLS_EXTENSION__: any;
  }
}
```

## 可见性
对于`declare namespace ABC`
- 如果声明文件没有 export 或 import 语句（即"脚本"模式），那么`ABC`会成为全局命名空间
- 如果声明文件包含 export 或 import，它就是一个模块，此时 declare namespace 不会自动暴露到全局，解决方法：
    - 在需要使用的文件中显式导入
    - 使用 declare global 包裹来强制提升到全局
        ```ts
        declare global {
            namespace ABC {
                export const foo: string;
            }
        }
        ```

# export 在 ts 中的扩展语法
## export as namespace MyLib
可使 MyLib 在全局可访问（用于cdn场景）

## export = xxx
对应于 commonJS 中的 `module.exports = xxx`，将整个模块作为单个对象导出。
```ts
// fs.d.ts
declare module 'fs' {
  function readFile(path: string): void;
  // ... 很多函数
  export = fs;  // 整体导出
}
```

## export as namespace
用于 **UMD 模块**，让模块在通过 `<script>` 标签引入时也能作为全局变量访问。
```ts
// lodash.d.ts (UMD 格式)
export = _;           // CommonJS 方式导出
export as namespace _; // 同时作为全局变量 _ 可用

// 使用时：
// 作为模块：import * as _ from 'lodash';
// 作为全局：_.debounce(...) （CDN 引入时）
```

## 三者的关系和区别

| 语法 | 模块系统 | 能否与 `export =` 共存 | 使用场景 |
|------|----------|----------------------|----------|
| `export = xxx` | CommonJS | - | Node.js 库 |
| `export as namespace X` | UMD | ✅ 可以 | 需要同时支持模块和全局的库 |
| `export const xxx` | ESM | ❌ 不可以 | 现代 ES 模块 |

### 组合使用示例（UMD 格式）
```ts
// my-lib.d.ts
export function foo(): void;
export const version: string;

export = myLib;              // CommonJS 导出
export as namespace myLib;   // 全局变量也可用

declare namespace myLib {
  export { foo, version };
}
```

### 错误示例
```ts
// ❌ 错误：export = 和 ESM export 不能同时出现
export = myLib;
export const foo = 1;  // 语法错误！
```

# 为什么 DefinitelyTyped 拒绝 declare module 'xxx'

## 原因
在 DefinitelyTyped 中，类型定义包的目录名（如 `@types/lodash` 中的 `lodash`）**已经被 TypeScript 自动识别为模块名**。如果在 `index.d.ts` 中再写 `declare module 'lodash'`，会导致：

1. **重复声明**：模块被定义两次
2. **路径嵌套错误**：使用时会出现 `lodash/lodash` 这样的错误嵌套路径

## 示例说明

### ❌ 错误写法（在 DefinitelyTyped 中）
```ts
// types/foo/index.d.ts
declare module 'foo' {  // ❌ 错误：目录名已经是 'foo'，这里再 declare 会造成嵌套
  export function bar(): void;
}
```
使用时会出现错误路径：
```ts
import { bar } from 'foo/foo';  // 错误地变成了 foo/foo
```

### ✅ 正确写法
```ts
// types/foo/index.d.ts
// 直接写类型定义，不要包裹 declare module
export function bar(): void;
export const VERSION: string;
```

## 何时可以使用 declare module
在以下场景中使用 `declare module` 是合理的：

1. **为无类型的第三方库补充类型**（不是发布到 @types 的包）
   ```ts
   // 项目内的 types.d.ts
   declare module 'untyped-lib' {
     export function doSomething(): void;
   }
   ```

2. **扩展已有模块的类型**（模块扩充/插件类型）
   ```ts
   // 为 vue 添加自定义属性
   declare module 'vue' {
     interface ComponentCustomProperties {
       $myProperty: string;
     }
   }
   ```

3. **声明通配符模块**（资源文件）
   ```ts
   declare module '*.png' {
     const src: string;
     export default src;
   }
   ```
