# registry

要在服务器上架设自己的 docker 镜像源，应该首先在服务器安装 podman，然后：

## 获取 registry 镜像

```sh
docker pull registry:2
```
如果服务器连不了外网，可以本地下载，然后搬到服务器上：
- docker save
- scp
- docker load

## 启动容器

```sh
podman run -d \
  --name registry \
  -p 0.0.0.0:15000:5000 \
  -v ~/registry/data:/var/lib/registry:Z \
  -v ~/registry/auth:/auth:Z \
  -e "REGISTRY_AUTH=htpasswd" \
  -e "REGISTRY_AUTH_HTPASSWD_REALM=Registry Realm" \
  -e "REGISTRY_AUTH_HTPASSWD_PATH=/auth/htpasswd" \
  registry:2
```

- --name: 指定容器名称
- -p: 相当于`--publish`，端口映射
    - 127.0.0.1: 只接受本地的连接
    - 第一个 15000: 宿主机端口
    - 第二个 5000: 容器内端口
- -v: 相当于`--volumn`，挂载卷，指定容器内的特定目录使用宿主机目录，持久化数据
    - -v ~/registry/data:/var/lib/registry:Z
        - ~/registry/data: 宿主机目录（镜像实际存储位置）
        - /var/lib/registry: 容器内目录（镜像默认存储位置）
        - 结尾的`:Z`: SELinux 标签，让容器可以读写该目录（Podman/Docker 的 SELinux 安全策略）
    - -v ~/registry/auth:/auth:Z 
        - 挂载认证文件目录，包含`htpasswd`用户名密码文件
- -e: 相当于`--env`
    - "REGISTRY_AUTH=htpasswd"，设置身份认证模式为`htpasswd`
    - "REGISTRY_AUTH_HTPASSWD_REALM=Registry Realm"，设置认证域名称，弹窗提示时会显示
    - "REGISTRY_AUTH_HTPASSWD_PATH=/auth/htpasswd"，指定`htpasswd`文件在容器内的路径

## 设置密码

安装 apache 工具包，然后使用其中的 `htpasswd` 工具追加文件到之前挂载的密码文件

```bash
# Debian/Ubuntu
sudo apt install apache2-utils

# CentOS/RHEL/Fedora
sudo dnf install httpd-tools

# 生成
htpasswd -Bbn admin 你的强密码 > ~/registry/auth/htpasswd
```

## 验证

```bash
curl -u admin:正确密码 http://127.0.0.1:5000/v2/_catalog  # 成功 200
curl -u admin:错密码 http://127.0.0.1:5000/v2/_catalog  # 失败 401
```

## 登录

如果是 http 访问，必须显式设置不安全的 registry 列表，否则无法登录

### desktop

在`设置-Docker Engine` 中添加：

```json
{
"insecure-registries": ["服务器IP:5000"]
}
```

### linux

vim /etc/containers/registries.conf

```toml
[[registry]]
location = "your-registry.example.com:5000"
insecure = true
```

### 登录

```bash
docker login IP:port
```

然后就可以进行 push 和 pull 操作了

#### 登录状态丢失

在一些老版的 podman 中，登录状态可能会过段时间就掉了

```sh
echo 'export REGISTRY_AUTH_FILE="$HOME/.config/containers/auth.json"' >> ~/.bashrc
```

使用环境变量指定登录状态存储位置，然后 source .bashrc，并重新登录，检查那个文件`auth.json`是否出现在指定位置

## 注意

- 推送进去的镜像并不能用 podman image ls 看到，因为它们存在于名为 registry 的容器中，与外面这个 podman 客户端是分开的
- 如果要推送镜像，镜像必须命名为 `host:port/namespace/image:tag`，这是规定，docker 会根据镜像里的 host 来决定往哪推，否则就推去 docker.io 了
- 多次推送同一名称的镜像，不会在 registry 产生 dangling 镜像，但会残留不再引用的 blob，可以用以下命令运行 GC
    ```sh
    podman exec -it registry:2 /bin/registry garbage-collect /etc/docker/registry/config.yml
    ```

## 配置 systemd 服务（开机自启）

为了让 registry 容器在系统启动时自动运行，可以创建一个 systemd 用户服务单元文件。

Podman 与 systemd 集成有两种主流方式：

### 方案 A：前台运行 + Type=notify（推荐）

使用 `--sdnotify=conmon`，让 conmon 作为前台进程运行，并通过 `sd_notify` 协议向 systemd 报告服务就绪状态。这是更现代的集成方式，配置更简洁。

```ini
# ~/.config/systemd/user/registry.service

[Unit]
Description=Podman container - Private Docker Registry
Documentation=https://docs.podman.io/
Wants=network-online.target
After=network-online.target
RequiresMountsFor=%h/registry/data %h/registry/auth

[Service]
# Type=notify: systemd 等待 podman 通过 sd_notify 报告就绪后才标记服务为 active
Type=notify
# NotifyAccess=all: 允许服务及其子进程发送 sd_notify 通知
NotifyAccess=all
Environment=PODMAN_SYSTEMD_UNIT=%n
Restart=always
RestartSec=5
TimeoutStartSec=300
TimeoutStopSec=70
ExecStartPre=-/usr/bin/podman rm -f registry
# 注意：没有 -d，容器在前台运行；--sdnotify=conmon 让 conmon 向 systemd 报告状态
ExecStart=/usr/bin/podman run --rm \
  --name registry \
  -p 0.0.0.0:15000:5000 \
  -v %h/registry/data:/var/lib/registry:Z \
  -v %h/registry/auth:/auth:Z \
  -e "REGISTRY_AUTH=htpasswd" \
  -e "REGISTRY_AUTH_HTPASSWD_REALM=Registry Realm" \
  -e "REGISTRY_AUTH_HTPASSWD_PATH=/auth/htpasswd" \
  --sdnotify=conmon \
  registry:2
ExecStop=/usr/bin/podman stop -t 10 registry
KillMode=mixed

[Install]
WantedBy=default.target
```

### 方案 B：后台运行 + Type=forking（兼容旧版本）

使用传统的 `-d` 后台运行方式，通过 `PIDFile` 让 systemd 追踪进程。兼容性更好，适用于不支持 `--sdnotify` 的老版本 podman。

```ini
# ~/.config/systemd/user/registry.service

[Unit]
Description=Podman container - Private Docker Registry
Documentation=https://docs.podman.io/
Wants=network-online.target
After=network-online.target
RequiresMountsFor=%h/registry/data %h/registry/auth

[Service]
# Type=forking: podman run -d 会 fork 出守护进程后退出，systemd 通过 PIDFile 追踪
Type=forking
PIDFile=%t/containers/registry.pid
Environment=PODMAN_SYSTEMD_UNIT=%n
Restart=always
RestartSec=5
TimeoutStartSec=300
TimeoutStopSec=70
ExecStartPre=-/usr/bin/podman rm -f registry
ExecStart=/usr/bin/podman run -d --rm \
  --name registry \
  -p 0.0.0.0:15000:5000 \
  -v %h/registry/data:/var/lib/registry:Z \
  -v %h/registry/auth:/auth:Z \
  -e "REGISTRY_AUTH=htpasswd" \
  -e "REGISTRY_AUTH_HTPASSWD_REALM=Registry Realm" \
  -e "REGISTRY_AUTH_HTPASSWD_PATH=/auth/htpasswd" \
  --conmon-pidfile %t/containers/registry.pid \
  --cidfile %t/containers/registry.id \
  registry:2
ExecStop=/usr/bin/podman stop -t 10 registry
# KillMode=none: 必须设置，否则 systemd 可能会误杀 podman 的子进程
KillMode=none

[Install]
WantedBy=default.target
```

### 两种方案对比

| 特性 | 方案 A (notify) | 方案 B (forking) |
|------|----------------|------------------|
| 运行方式 | 前台运行，无 `-d` | 后台运行，`-d` |
| 进程追踪 | systemd 直接追踪前台进程 | 通过 `PIDFile` 间接追踪 |
| 就绪通知 | `sd_notify` 精确报告 | 依赖 PID 文件存在 |
| KillMode | `mixed`（正常行为） | `none`（必须设置，防止误杀） |
| 配置复杂度 | 更简洁 | 需要 `PIDFile`、`--conmon-pidfile`、`--cidfile` |
| 兼容性 | 需要 podman ≥ 3.0 | 兼容所有版本 |

### 启用步骤

```bash
# 1. 创建 systemd 用户服务目录
mkdir -p ~/.config/systemd/user

# 2. 将上面的内容保存到 ~/.config/systemd/user/registry.service
# 3. 重新加载 systemd 配置
systemctl --user daemon-reload

# 4. 启动服务
systemctl --user start registry

# 5. 设置开机自启
systemctl --user enable registry

# 6. 查看服务状态
systemctl --user status registry
```

### 关键配置说明

- `%h`: systemd 变量，表示用户的家目录（如 `/home/username`）
- `%t`: systemd 变量，表示运行时目录（如 `/run/user/1000`）
- `%n`: systemd 变量，表示当前单元文件的完整名称
- `-` 前缀: 表示命令执行失败也不会导致服务启动/停止失败
- `--rm`: 容器停止后自动删除，配合 systemd 的 `Restart=always` 实现"每次重启都是全新容器"的最佳实践
- `--sdnotify=conmon`: 让 conmon 通过 systemd 通知协议报告容器就绪状态（方案 A 关键配置）
- `--conmon-pidfile` 和 `--cidfile`: 让 podman 生成 PID 和容器 ID 文件，便于 systemd 追踪（方案 B 需要）
