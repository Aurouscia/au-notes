# ODS（Operational Data Store）运营数据层 
- 原始数据“冷冻室” 
- 把源系统数据 1:1 镜像到数仓，保留完整历史，不做业务逻辑 
- 与源表字段同名同类型，加 `__load_time` 等审计列；可按天分区、压缩、冷备；不建主键 
- 不应：把 ODS 当“临时区”第二天就删；或过早做字段清洗，导致溯源困难

# DWD（Data Warehouse Detail）明细数据层 
- 业务过程“原子事实” 
- 以“业务过程+粒度”为核心，做统一清洗、编码转换、维度退火，得到干净明细
- 范式或星型混合，统一主键（如 `order_id`）、统一时间戳、统一码值；可分区可分桶；冗余常用维度字段（维度退火）
- 不应：把 DWD 做成“大宽表”导致粒度丢失；或继续保留源系统码值，造成下游口径不一致

# DWS（Data Warehouse Summary）汇总数据层
- 公共汇总“半成品”
- 把 DWD 按“维度+业务过程”汇总到日/周/月，解决 80% 重复聚合需求
- 星型模型，以“统计粒度”为主键（如 `shop_id, sku_id, dt`）；指标可累加；可加轻度维度字段
- 不应：把 DWD 所有维度都汇总进去变成“巨宽表”；或加不可累加指标（去重）导致误用

# DM（Data Mart）数据集市层
- 部门级“即食面”
- 面向部门/场景做高度业务化封装，屏蔽底层表，直接供 BI、AI、API 调用
- 大宽表、拉链表、cube 均可；允许冗余 100% 维度；主键=业务需求主键；可加派生指标
- 不应：把 DM 当“万能层”继续向下游出宽表，造成重复建设；或反向调用 ODS，破坏层级

# DIM（Dimension）维度层
- 跨层共享“字典”
- 统一所有层级要用到的描述性字段，保证“同一个维度，同一个版本”
- 星型维度表、拉链表、快照表；主键稳定（如 `sku_id`）；可加缓慢变化维 SCD2；可被 ODS/DWD/DWS/DM 同时引用
- 不应： 把维度字段冗余到每一层且各层自己维护，造成“同名不同值”；或 DIM 表源系统一改就全量失效
